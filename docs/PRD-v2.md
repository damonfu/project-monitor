## 1. 产品重新定位

### 1.1 产品愿景

从"项目展示"升级为"监控与行动闭环平台"。

**原定位**：本地目录 + Git 驱动的项目状态可视化平台
**新定位**：智能项目健康监控 + 主动告警 + 一键行动的闭环平台

### 1.2 目标用户

| 用户类型 | 描述 | 核心需求 |
|---------|------|----------|
| 技术负责人 | 管理多个项目的团队lead | 快速掌握所有项目状态、及时发现风险 |
| 独立开发者 | 维护多个个人项目 | 低成本监控、自动化日报 |
| 自动化 Bot | coder_bot、其他Agent | 标准化API、可编程接口 |

### 1.3 核心价值

1. **信息聚合**：多仓库状态 → 单页总览
2. **主动告警**：被动查看 → 主动推送（未提交、不活跃、异常）
3. **行动闭环**：发现问题 → 推送通知 → 快速处理
4. **自动化提效**：定时扫描、自动报告、无需人工介入

### 2.2 功能详细说明

#### F1. 告警系统（P0/P1）

**目标**：主动发现问题，及时通知用户。

**告警规则**：
- **未提交告警**：工作区有未提交更改超过 N 小时
- **不活跃告警**：项目超过 N 天没有提交
- **异常告警**：扫描失败、配置错误等

**告警状态**：
- `active`：活跃告警，需要处理
- `acknowledged`：已确认，处理中
- `ignored`：已忽略，不再提醒

**前端展示**：
- 告警列表页：展示所有告警，支持筛选（类型、状态、项目）
- 告警详情页：展示告警详情、历史记录、相关项目信息
- 告警操作：确认、忽略、查看项目

**通知渠道**：
- 飞书群消息（P1）
- 邮件通知（P2）
- Webhook（P2）

#### F2. 项目管理（P1）

**目标**：支持项目分组、收藏、个性化配置。

**项目分组**：
- 按标签分组：backend、frontend、tools 等
- 按目录分组：自动识别 ~/Codes 下的目录结构
- 自定义分组：用户手动创建分组

**项目收藏**：
- 快速访问收藏的项目
- 收藏项目优先展示

**项目配置**：
- 告警阈值配置（未提交时间、不活跃天数）
- 告警开关（启用/禁用）
- 自定义标签

#### F3. 定时任务（P0）

**目标**：自动化扫描、快照、报告生成。

**任务列表**：
- 每日扫描：定时扫描项目状态
- 快照生成：生成当日快照
- 报告生成：生成日报
- 告警检查：检查告警规则，生成告警

**触发方式**：
- **方案 A**：Cloudflare Cron Trigger（推荐）
- **方案 B**：coder_bot 定时任务
- **方案 C**：本地 crontab

**执行频率**：
- 扫描：每 30 分钟
- 快照：每日 1 次（凌晨）
- 报告：每日 1 次（凌晨）
- 告警检查：每 30 分钟

## 4. 技术方案建议

### 4.1 前端 API 集成方案

**当前问题**：前端使用 mock 数据，未连接真实 API。

**解决方案**：分层渐进式集成。

**步骤**：
1. **创建 API 层**：`src/services/api.ts`
   - 封装 fetch 请求
   - 处理错误和重试
   - 添加缓存机制

2. **创建数据层**：`src/hooks/useProjects.ts` 等
   - 使用 React Query 或 SWR
   - 管理数据状态
   - 处理缓存和刷新

3. **替换 mock 数据**：
   - 项目列表页：替换 mock 数据
   - 项目详情页：替换 mock 数据
   - 报告页面：替换 mock 数据

4. **添加加载和错误状态**：
   - 加载中：骨架屏或加载动画
   - 加载失败：错误提示和重试按钮
   - 离线：离线提示

**验收指标**：
- 首页加载时间 < 2s
- API 请求成功率 > 99%
- 错误处理覆盖率 100%

### 4.2 定时任务接入方案

**当前问题**：扫描、快照、报告生成需要手动执行。

**方案对比**：

| 方案 | 优点 | 缺点 | 推荐度 |
|------|------|------|--------|
| Cloudflare Cron Trigger | 无需服务器、稳定可靠 | 需要 Workers 付费计划 | ⭐⭐⭐⭐⭐ |
| coder_bot 定时任务 | 已有基础、易于调试 | 依赖外部服务 | ⭐⭐⭐⭐ |
| 本地 crontab | 简单直接 | 依赖本机运行 | ⭐⭐⭐ |

**推荐方案**：Cloudflare Cron Trigger

**实现步骤**：
1. 在 `wrangler.toml` 中配置 Cron Trigger

2. 在 Worker 中实现定时任务

3. 部署并测试

**验收指标**：
- 定时任务执行成功率 > 99%
- 扫描时间 < 30s
- 告警检查时间 < 10s

### 4.3 移动端适配方案

**当前问题**：基础响应式已完成，但可能不够完善。

**优化策略**：
1. **布局优化**：单列布局、折叠内容区域、固定底部导航
2. **交互优化**：大按钮、大字体、手势支持
3. **性能优化**：图片懒加载、数据分页加载、缓存策略

**验收指标**：
- 移动端 FCP < 1.5s
- 移动端 LCP < 2.5s
- 移动端 CLS < 0.1

## 5. 用户体验设计建议

### 5.1 整体交互流程

用户进入平台 → 查看项目总览 → 可选择查看项目详情/告警列表/日报列表 → 进行相应操作

### 5.2 关键页面设计

#### 项目总览页
- 顶部：标题 + 统计卡片 + 刷新按钮
- 中部：项目列表（卡片/表格切换）
- 底部：快速操作

#### 告警列表页
- 顶部：筛选器（类型、状态、项目）
- 中部：告警列表
- 底部：批量操作

#### 项目详情页
- 顶部：项目信息 + 操作按钮
- Tab 1：概览
- Tab 2：提交历史
- Tab 3：文件变更
- Tab 4：告警

### 5.3 告警和通知设计

#### 告警分级

| 级别 | 颜色 | 说明 | 示例 |
|------|------|------|------|
| Critical | 红色 | 严重问题，需立即处理 | 工作区脏 > 48h |
| Warning | 黄色 | 警告，需要注意 | 工作区脏 > 24h |
| Info | 蓝色 | 信息提示 | 项目不活跃 > 7d |

#### 通知降噪策略
- 聚合通知：同类型告警聚合后推送
- 静默期：确认后的告警进入静默期（24h）
- 去重：相同告警只推送一次

## 6. 风险评估与应对

### 6.1 风险矩阵

| 风险 | 概率 | 影响 | 应对策略 |
|------|------|------|----------|
| 前端 API 集成复杂度高 | 中 | 高 | 分层渐进式集成 |
| 定时任务稳定性问题 | 低 | 高 | 添加重试机制、监控告警 |
| 移动端适配工作量大 | 中 | 中 | 优先核心功能，逐步优化 |
| 告警系统误报 | 中 | 中 | 优化规则、添加静默期 |
| 用户需求变化 | 高 | 中 | 敏捷迭代、快速响应 |

### 6.2 应对措施

1. **前端 API 集成**：使用渐进式集成，先替换核心页面
2. **定时任务稳定性**：添加重试机制（最多 3 次）、记录执行日志
3. **移动端适配**：优先保证核心功能可用，逐步优化非核心功能
4. **告警误报**：优化告警规则、添加静默期机制
5. **用户需求变化**：采用敏捷迭代，每周发布一个小版本
